**Date**: 12 de marzo de 2026  
**Status**: Design Phase 1 Complete

## Entity: Departamento

### JPA Entity Definition

```java
@Entity
@Table(name = "departamentos", 
    uniqueConstraints = { @UniqueConstraint(columnNames = "nombre") })
public class Departamento {
    
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @NotEmpty(message = "Nombre is required")
    @Size(min = 1, max = 255, message = "Nombre must be between 1 and 255 characters")
    @Column(nullable = false, length = 255)
    private String nombre;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false)
    private EstadoAcceso estado = EstadoAcceso.ACTIVO;

    @CreationTimestamp
    @Column(nullable = false, updatable = false)
    private LocalDateTime creadoEn;

    @UpdateTimestamp
    @Column(nullable = false)
    private LocalDateTime actualizadoEn;

    // Standard getters/setters
}
```

### Attributes

| Attribute | Type | Constraints | Description | Validation |
|-----------|------|-------------|-------------|-----------|
| `id` | Long | PK, Auto-increment | Unique identifier | Generated by DB |
| `nombre` | String | NOT NULL, UNIQUE, VARCHAR(255) | Department name | Required, 1-255 chars, unique in table |
| `estado` | Enum(ACTIVO/INACTIVO) | NOT NULL, Default ACTIVO | Lifecycle state | Must be valid enum value |
| `creadoEn` | LocalDateTime | NOT NULL, Immutable | Creation timestamp | Auto-populated on insert |
| `actualizadoEn` | LocalDateTime | NOT NULL | Last update timestamp | Auto-updated on any change |

### Validation Rules

#### At Entity Level (javax.validation)
- `@NotEmpty(nombre)` - Name cannot be empty string
- `@Size(min=1, max=255)` - Name must be 1-255 characters
- `@NotNull(estado)` - Estado cannot be null
- Default value: `estado = EstadoAcceso.ACTIVO` when creating new record

#### At Database Level (Flyway SQL)
- `UNIQUE(nombre)` - No duplicate department names
- `NOT NULL` constraints on id, nombre, estado, creadoEn, actualizadoEn
- `VARCHAR(255)` for nombre column
- Primary key on id
- Indexes on nombre (for UNIQUE constraint)

#### At Application Level (Service)
- Check for duplicate nombre before insert (catch `DataIntegrityViolationException` → HTTP 409)
- Check for associated employees before delete (query Empleado table → HTTP 409 if any found)
- Filter out INACTIVO records in GET/LIST operations (WHERE estado = 'ACTIVO')
- Enforce estado ACTIVO when reading records (404 if INACTIVO)

### Referential Integrity

#### Soft-Delete Policy
- **Operation**: DELETE request marks record as INACTIVO instead of hard-deleting
- **Impact**: INACTIVO records not returned in normal queries
- **Recovery**: Could be reactivated via direct SQL update if needed (audit trail intact)
- **Future**: When Empleado adds FK to Departamento, treat as advisory-only (department can be INACTIVO while employees exist)

#### Employee Association Check (Before Delete)
```java
// In DepartamentoService.delete(id):
if (empleadoRepository.existsByDepartamentoId(id)) {
    throw new ConstraintViolationException("Cannot delete department with associated employees");
    // Returns HTTP 409 CONFLICT
}
// If no employees, proceed with soft-delete (update estado = INACTIVO)
```

## Service Layer

### DepartamentoService

**Create**:
1. Validate input (name not empty, ≤ 255 chars)
2. Check for duplicate nombre (query by name + ACTIVO status)
3. If duplicate found → HTTP 409 CONFLICT
4. Create entity with estado = ACTIVO
5. Persist to DB
6. Return HTTP 201 + DepartamentoResponse

**Read**:
1. Query by ID and estado = ACTIVO
2. If not found (ID doesn't exist or is INACTIVO) → HTTP 404 NOT FOUND
3. Return HTTP 200 + DepartamentoResponse

**List**:
1. Query all records WHERE estado = ACTIVO, paginated 10 per page
2. Return HTTP 200 + Page<DepartamentoResponse>

**Update (PATCH)**:
1. Query by ID and estado = ACTIVO
2. If not found → HTTP 404
3. Validate new nombre (length check, not empty)
4. Check for duplicate nombre (excluding current record)
5. If duplicate found → HTTP 409
6. Update entity
7. Return HTTP 200 + updated DepartamentoResponse

**Delete (Soft)**:
1. Query by ID and estado = ACTIVO
2. If not found → HTTP 404
3. Check if any employees associated (FK foreign key check)
4. If employees exist → HTTP 409 CONFLICT
5. Mark entity as estado = INACTIVO
6. Persist changes
7. Return HTTP 204 NO CONTENT

## Error Handling

### HTTP Status Codes

| Code | Scenario | Example |
|------|----------|---------|
| 400 | Validation error | NOME empty, name > 255 chars, null values |
| 401 | No authentication | Missing/invalid Basic Auth header |
| 404 | Not found | ID doesn't exist or is marked INACTIVO |
| 409 | Conflict | Duplicate name, or delete with employees |
| 500 | Server error | Unhandled exception in service |

## Design Decisions Rationale

### D1: Why Soft-Delete Instead of Hard-Delete?
- **Benefit**: Maintains audit trail, enables recovery, aligns with empleados pattern
- **Trade-off**: Requires filtering on stato = ACTIVO in all queries, uses more storage
- **Mitigation**: Implement systematic filtering in repository, document in architecture guide

### D2: Why @UniqueConstraint at Both DB and APP Level?
- **DB Level**: Enforces at integrity layer, prevents race conditions in concurrent inserts
- **APP Level**: Catches violations early, enables graceful error response with 409 status
- **Trade-off**: Slight additional code complexity
- **Mitigation**: Standardize pattern across codebase

### D3: Why Prevent Delete Instead of Cascade?
- **Benefit**: Protects employee records, maintains data integrity, prevents cascading data loss
- **Trade-off**: User must manually reassign employees before deleting department
- **Mitigation**: Document workflow, provide clear error messages suggesting reassignment step

### D4: Why Reuse EstadoAcceso Enum?
- **Benefit**: Consistency across entities, reduces types in codebase, facilitates future shared reference table
- **Trade-off**: Couples Departamento to empleados module (soft coupling via enum)
- **Mitigation**: Enum is stable and unlikely to change; document intent in code comments

## Future Enhancement Opportunities

1. **Full Soft-Delete with Restore**: Add endpoint PATCH `/api/v1/departamentos/{id}/restore` to reactivate INACTIVO departments
2. **Foreign Key to Empleado**: Add `departamento_id` FK in empleados table (future feature)
3. **Audit Logging**: Add `modificado_por` (usuario que hizo el cambio) and audit table for compliance
4. **Filtering by Estado**: Add query parameter `?estado=ACTIVO` to explicitly request INACTIVO departments if needed
5. **Batch Operations**: Add `/api/v1/departamentos/bulk-update` or `/bulk-delete` for admin operations
